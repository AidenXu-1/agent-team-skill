# Agent Team 读取路由优化清单

> 目的:把多会话协作的阅读边界从“靠 Agent 自觉”改成“短入口 + 固定脚本 + 结构化元数据 + 触发式读正文”。

## 会话模式

- 只保留两种模式:自动模式 / 人工模式。
- 自动模式:Agent 具备创建会话、命名会话、发送短唤醒的会话管理能力。
- 人工模式:Agent 不声称已通知其他部门;只写目标部门收件箱,并提醒用户手动通知对应部门。
- 置顶、排序不是必要能力,不作为模式判断条件。

## 上岗引导

- `上岗引导.md` 是轻量常态化路由卡,不是长制度手册。
- 它只说明当前部门身份、层级、先运行哪个读取脚本、默认读什么、默认不读什么、自动/人工通知方式。
- 长规则放在共享规则文档、岗位说明、收件箱模板和报告模板中,不要在每个上岗引导里重复堆叠。

## 读取路由脚本

- scaffold 生成 `docs/collaboration/scripts/agent_team_read.py`。
- 脚本是读取路由器,不是管理大脑。
- 脚本只做稳定裁剪:
  - 根据部门名返回身份、层级、通知模式、会话 ID。
  - 返回本部门必读文件路径。
  - 返回收件箱当前待办标题。
  - 读取 Markdown frontmatter,返回 `summary` 等人工预写字段。
  - 按 `type`、`department`、`target`、`status`、`tags` 查候选文件。
- 脚本不做创造性总结、不替代统筹判断、不替代审核结论、不自动放行。

## Markdown 元数据

- 普通 Markdown 的 YAML frontmatter 不是平台级索引;不能假设 Agent 会稳定只读 YAML。
- 报告、审核报告、专项结论、关键决策都使用统一基础字段:

```yaml
---
type: audit_report
department: 测试部
target: IndexTTS2分段生成
status: blocked
date: 2026-07-02
related_task: 测试部收件箱#2026-07-02-001
decision: 不通过
tags: [用户可见出口, 错误文案, worker, UI]
summary: 用户最终错误文案丢失段号，底层通过不代表出口通过。
---
```

`type` 可用值:

- `work_report`
- `audit_report`
- `special_conclusion`
- `decision_record`

## 报告体系

- 不是所有部门都必须产出正式报告。
- 所有部门默认维护收件箱回报、交接班文档、日志索引和必要产出路径。
- 审核层必须产出审核报告。
- 执行层只有遇到复杂研究、设计、方案、架构、数据分析时才产出工作报告。
- 统筹部只有需要阶段总结、决策材料、用户判断材料时才产出报告。
- “把关报告”在规则中统一解释为审核层报告;目录可兼容旧名,模板标题应写“审核报告”。

## 专项结论

- 只影响一个任务、一个部门:放在对应报告正文。
- 会被多个部门复用:升格到 `docs/collaboration/专项结论/`。
- 长期改变项目规则、架构、依赖、安全或发布方式:升级到 `docs/decisions/`。

## 默认阅读边界

- 默认读:上岗引导、读取路由脚本输出、岗位说明、交接班文档、收件箱、必要错题摘要。
- 默认不读:日志正文、报告正文、决策正文、其他部门正文、代码 diff、测试证据全文。
- 触发才读正文:摘要不足、路径异常、结论冲突、涉及放行/返工/安全/费用/发布/用户可见质量、用户要求查证据、当前任务明确依赖正文。

## 进度与决策

- `docs/progress.md` 是项目级总状态,由统筹部维护。
- 统筹部接班默认读;新增部门/新会话上岗、任务背景不足、用户问整体进度、阶段切换、收口、发布前、多部门冲突时才读。
- 执行层和审核层平时不默认读 `docs/progress.md`。
- 重大技术决策双写但不重复正文:部门日志写一行索引,`docs/decisions/` 写正式决策正文。
